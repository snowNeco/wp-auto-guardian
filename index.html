Pain

“保守が回らない”のは、担当者の責任じゃない。

Web制作会社の保守は、人手・目視・担当者依存になりがちです。 その結果「事故が起きてから気づく」「保守が赤字化する」構造が生まれます。

更新のたびに全ページ目視

プラグイン更新→表示崩れが怖い→人が巡回。工数が膨らみ赤字に。

容量・バックアップ異常に後から気づく

ディスク逼迫／バックアップ失敗／更新放置が積み上がり、事故は“いつか”起きる。

属人化して引き継げない

担当者の頭の中にしか運用がなく、休みや退職で一気に崩れる。

誠実でいたいのに、工数が足りない

理想はある。だが現実は案件が詰まり、保守が後回しになる。

Solution

「目視」を「仕組み」に変える、2つの自動化エンジン

SiteOps Guard は、表示チェックと保守監視を分離し、毎日回る運用フローとして実装します。

Engine 01

Visual Guard（表示テスト）

Playwright がページを巡回し、ピクセル差分で崩れを検知。人の“見落とし”を前提から消します。

  • PC / スマホ / タブレットの3デバイス比較
  • 差分レポート（どこがズレたか）を自動生成
  • 更新前後の比較で「更新して良い」が即判断可能
Engine 02

Server Watcher（保守監視）

GitHub Actions が 24/365 で状態を収集し、異常を先に通知。事故が“起きる前”に手を打てます。

  • ディスク残量、バックアップ生存、更新状況を自動チェック
  • Chatwork / Slack / Teams 等へ通知（運用に合わせる）
  • 確認の“ルーチン”を、担当者の努力から切り離す
Features

導入後に変わるのは「作業時間」だけじゃない。

事故を減らし、保守を回し、提案に時間を戻す。保守部門の収益構造を変えます。

判断はディレクター、作業は仕組み

“判断が必要な箇所”だけ人が見れば良い状態に落とし込みます。

  • 「異常の有無」を先に絞り込む
  • 若手・事務でも運用できる手順へ分解
  • 属人化しないチェックリストと通知

保守の可視化（= 価値の説明ができる）

保守は“やって当たり前”にされがち。レポートで価値が伝わるようになります。

  • 実施内容が残る（監視・検知・対応履歴）
  • 顧客説明が容易（保守費の納得感）
  • 提案につながる（改善点が見える）
Package

「ツール」ではなく、「運用フロー」を納品します。

スクリプトだけ渡して終わり、にしない。現場で回る形まで落とし込みます。

提供物

  • Visual Guard（表示テスト）一式
  • Server Watcher（監視レポート）一式
  • 通知連携（Chatwork等）設定
  • 運用手順書（引き継ぎ可能な形）

“現場目線”の設計

設計者が50サイト以上の保守運用を経験しているから、机上の理想論になりません。

  • 制作会社の運用導線（担当・権限・時間）を前提にする
  • “やりっぱなし”にならない通知と判断導線
  • 導入後のオペレーションが崩れない設計
Impact

導入後、保守運用はこう変わります。

制作会社の保守運用を前提にした導入検証ベース想定です（運用規模・対象範囲により前後します）。

① 保守確認工数

約80%削減

  • 全ページ目視巡回 → 差分検知箇所のみ確認
  • 管理画面巡回 → 自動レポート確認
  • 手動チェック → 通知ベース確認

② 事故発生リスク

事前検知化

  • 容量不足・バックアップ異常を自動監視
  • 更新放置・崩れを自動検知
  • 「知らなかった」を構造的に排除

③ 担当者依存

属人化解消

  • 確認作業を仕組みに移管
  • 若手・事務でも運用可能
  • ディレクターは判断のみ

④ 保守収益性

赤字 → 安定利益

  • 工数削減により粗利改善
  • 保守説明が容易（価値が伝わる）
  • 提案・改善の時間が戻る

更新作業フローの変化例

導入前

  • プラグイン更新
  • 全ページ目視巡回（30〜60分）
  • 担当者の記憶依存

導入後

  • プラグイン更新
  • 差分通知確認（3〜5分）
  • 異常箇所のみ確認

※目安：30〜50サイト規模の制作会社保守運用を想定。対象ページ数・監視項目・通知設計により変動します。

Pricing

料金は「導入＋運用」の構成です。

対象サーバー数・運用体制・監視項目により最適が変わるため、LPでは概要のみ掲載しています。
詳細単価（追加SSH設定など）は資料にまとめています。

導入

初期構築・運用定着

  • サーバー設定代行（初回）
  • SSH / Secrets 設定、認証、初期テスト
  • GitHub連携・実行環境の整備
  • オンボーディング（運用レクチャー・初期サポート）

※環境差分（例：Xサーバー種別）により追加設定が発生する場合があります。

運用

運用サポート（任意）

  • 月次チェック（定期確認）
  • 軽微な緊急対応
  • 通知ノイズの調整（必要に応じて）
フル管理

フルマネージド（任意）

  • 24/365監視・優先対応
  • 月次レポート
  • 運用の継続改善
